• I updated PRISM.ipynb, added a generic worker at PRISM/_multi_gpu_stage_worker.py:1, and patched PRISM/prism/runtime/
  assemble.py:35 plus PRISM/prism/profiling/meta_learner.py:143.

  What changed:

  1. Analysis and mode selection

  - The notebook now detects torch.cuda.device_count() and prints both T4s plus VRAM.
  - It estimates model weight size from an empty HF config, then does a single-GPU load probe.
  - Decision:
      - fits one T4 -> task_parallel
      - does not fit / probe fails -> model_parallel

  2. Case 1: task parallel on 2 GPUs

  - The old shard split was a simple midpoint split. I replaced it with balanced two-worker execution.
  - Profiling shard generation now runs in parallel through isolated GPU workers.
  - I also added helpers for later Stage 1 profile build and Stage 3 RTN precompute, again split by layer size, not
    blind full-work duplication.

  3. Case 2: model parallel across 2 GPUs

  - Added load_model_model_parallel(... device_map="auto" ...) in the notebook helper cell.
  - Fixed profiling so the MLP stats move correctly between sharded layer devices and the MLP device.
  - Fixed runtime assembly so it can assemble in place on a sharded base model with copy_model=False, and each wrapped
    layer stays on its original GPU.

  4. Assignment stage

  - I left assignment single-process. It is solver-heavy, but not the main GPU bottleneck. The GPU-sensitive parts are
    profiling and RTN precompute.
